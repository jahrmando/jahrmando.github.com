Failed to init inotify - Demasiados ficheros abiertos

01 Apr 2014

Recientemente se me han presentado estos eventos en un servicio SMB, haciendo la tarea me he encontrado con su respectiva razón y solución a estos eventos. Estos eventos son lanzados por el modulo inotify, que nos proporciona un mecanismo para monitorear eventos en el sistema de archivos.

Apr  1 12:50:57 srv smbd[27909]: [2014/04/01 12:50:57.200596,  0] smbd/notify_inotify.c:302(inotify_setup)
Apr  1 12:50:57 srv smbd[27909]:   Failed to init inotify - Demasiados ficheros abiertos
Apr  1 12:50:59 srv smbd[27909]: [2014/04/01 12:50:59.198617,  0] smbd/notify_inotify.c:302(inotify_setup)
Apr  1 12:50:59 srv smbd[27909]:   Failed to init inotify - Demasiados ficheros abiertos
Apr  1 12:51:01 srv smbd[27909]: [2014/04/01 12:51:01.212549,  0] smbd/notify_inotify.c:302(inotify_setup)
Apr  1 12:51:01 srv smbd[27909]:   Failed to init inotify - Demasiados ficheros abiertos
Apr  1 12:52:06 srv smbd[27909]: [2014/04/01 12:52:06.727637,  0] smbd/notify_inotify.c:302(inotify_setup)
Apr  1 12:52:06 srv smbd[27909]:   Failed to init inotify - Demasiados ficheros abiertos
Apr  1 12:52:18 srv smbd[27909]: [2014/04/01 12:52:18.989593,  0] smbd/notify_inotify.c:302(inotify_setup)
Apr  1 12:52:18 srv smbd[27909]:   Failed to init inotify - Demasiados ficheros abiertos

Cada instancia inotify esta asociado con una “watch list” dentro de esta cada elemento ‘watch’ tiene asociado un directorio o archivo para su monitoreo, para limitar el uso de recursos del sistema por parte de este modulo posee dos variables para esto: ‘/proc/sys/fs/inotify/max_user_instances’ y ‘/proc/sys/fs/inotify/max_user_watches’ y estos pueden ser incrementados a conveniencia.

Citando el manual de inotify:

/proc/sys/fs/inotify/max_user_instances

This specifies an upper limit on the number of inotify instances that can be created per real user ID.

/proc/sys/fs/inotify/max_user_watches

This specifies an upper limit on the number of watches that can be created per real user ID.

SMB hace uno del modulo inotify para monitorear los archivos que los usuarios usan, por ejemplo: el registro de actividades de los usuarios podemos monitorear que archivos ha leido, modificado o eliminado.

Pues si presentamos estos eventos antes mencionados es porque SMB esta creando mas instancias inotify por PID que las permitidas, esto pues se origina por la carga del servicio y exigencias de archivos por parte de los usuarios.

La solución esta en incrementar estos valores adecuando al nivel de peticiones.

Reviso el valor asignado a estas variables.

$ cat /proc/sys/fs/inotify/max_user_instances
128

$ cat /proc/sys/fs/inotify/max_user_watches
8192

Incremento el valor de las instancias permitidas por usuario que a mi parecer es demasiado bajo.

$ vim /etc/sysctl.conf
fs.inotify.max_user_instances = 256

Cargamos las configuraciones con sysctl.

$ sysctl -p

Verificamos que el valor este aplicado.

$ cat /proc/sys/fs/inotify/max_user_instances
256

En mi caso fue suficiente incrementar al valor 256 de max_user_instances y deje de presentar estos eventos.

Si quieres saber mas sobre como funciona inotify puedes hecharle un vistazo al proyecto python pyinotify
